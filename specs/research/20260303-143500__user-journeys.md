# Moontide AI - Product User Journeys (Sectioned)

## 1. Onboarding

### 1.1 New user signup flow

- User lands on app and clicks `Sign up`.
- User enters name, email, password.
- System creates user, creates/joins workspace context, signs user in.
- User lands on authenticated home.

### 1.2 Returning user login flow

- User opens app and clicks `Login`.
- User enters credentials.
- System restores session and workspace context.
- User lands on home with current state.

### 1.3 Integration setup flow (first-time)

- User opens setup/integrations.
- User connects GitHub and authorizes installation/org access.
- User connects Slack workspace.
- System validates both integrations and shows connected state.

### 1.4 Onboarding failure/recovery sub-flows

- Invalid credentials on login -> inline error -> retry.
- GitHub install missing repo/org scope -> reconnect/update install.
- Slack auth revoked/expired -> reconnect and refresh tokens.
- User signs out from settings -> session cleared -> redirected to login.

## 2. Agent Creation

### 2.1 Create from intent flow

- User opens `Agent Factory` and starts `Create Agent`.
- User provides name + intent.
- System generates initial draft config.
- User reviews typed config fields and creates draft agent.

### 2.2 Manual configuration flow

- User sets objective and instructions.
- User sets runtime mode (`slack_thread` or `background_triggered`).
- User selects repository scope (allowlisted repos).
- User configures trigger rules where applicable.
- User configures tooling and permission policy.
- User saves changes.

### 2.3 Activation flow

- User clicks `Activate`.
- System runs preflight validation:
- integrations connected
- repo scope selected
- required typed config fields valid
- approval channel present when policy needs it
- On pass: agent becomes active.
- On fail: user sees blockers and fixes them.

### 2.4 Creation failure/recovery sub-flows

- Draft generation timeout/error -> retry generation or manual edit.
- Missing repo selection -> save draft allowed, activation blocked.
- Invalid policy/tooling configuration -> inline validation errors.

## 3. Mention Agents (Interactive)

### 3.1 Start thread flow

- User mentions app in Slack root message.
- System creates thread-scoped runtime session.
- Agent posts first reply in that thread.

### 3.2 Multi-user conversation flow

- Any thread participant sends follow-up messages.
- All messages route to same agent session/sandbox for that thread.
- Agent replies stream/post back into same thread.

### 3.3 Agent selection flow (when multiple agents exist)

- User starts thread mention.
- System requires explicit agent selection in-thread.
- After selection, all thread messages bind to that chosen agent.

### 3.4 Control/permission sub-flows

- User sends terminate command (`@App terminate`) -> sandbox/session closed.
- Agent requests permission (`ask`) -> approval posted -> approve/deny.
- Approved permissions persist for future runs of same agent.

### 3.5 Mention failure/recovery sub-flows

- Thread has no selected agent -> prompt for selection.
- Session/sandbox dies mid-thread -> system reports error and offers restart path.
- Delivery failure to Slack thread -> run marked failed with reason in Mission Control.

## 4. Background Agents

### 4.1 PR-opened trigger flow

- GitHub `pull_request opened` event arrives.
- Matching rules resolve eligible active agents.
- System enqueues run(s) with dedupe keying.
- Worker executes run and writes artifact/outcome.

### 4.2 Scheduled trigger flow

- Cron schedule fires.
- Matching active agent run is enqueued.
- Worker runs against configured repo scope and policy.

### 4.3 Slack on-demand trigger flow

- User issues strict Slack trigger command.
- System validates agent + trigger eligibility.
- Run enqueued for async execution.

### 4.4 Queueing/retry/delivery sub-flows

- Agent already running -> new runs queue FIFO per agent.
- Retry policy applies on retryable failures.
- Overflow policy applies when queue limit is hit.
- Final output delivered to Mission Control and configured external targets.

### 4.5 Background failure/recovery sub-flows

- Deduped event -> duplicate dropped with traceable reason.
- Trigger misconfiguration -> run rejected with validation error.
- Runtime failure -> failed status + retry/terminate controls in Mission Control.

## 5. Update / Manage Agents

### 5.1 Edit and version flow

- User opens existing agent from factory list.
- User updates typed config (scope/instructions/triggers/policy).
- System saves and increments config version.

### 5.2 Lifecycle management flow

- User activates inactive agent.
- User deactivates active agent.
- User archives unused agent.

### 5.3 Integration/scope maintenance flow

- User reconnects GitHub/Slack if broken.
- User resyncs repo inventory from GitHub integration.
- User updates allowed repositories for each agent.

### 5.4 Management failure/recovery sub-flows

- Integration drift (tokens revoked/app removed) -> health shows degraded, user reconnects.
- Agent update invalidates activation prerequisites -> activation blocked until fixed.

## 6. Observability

### 6.1 Mission Control run visibility flow

- User opens runs list.
- User filters by agent, trigger type, status, time window.
- User opens run detail for timeline/events/errors/artifacts.

### 6.2 Operational actions flow

- User retries failed runs.
- User terminates active runs.
- User confirms final terminal state in timeline/status.

### 6.3 Health and approval visibility flow

- User sees integration health and connectivity issues.
- User sees pending permission requests and outcomes.
- User sees why a run failed (config/runtime/external dependency).

### 6.4 Observability failure/recovery sub-flows

- Missing external delivery -> visible as delivery-stage failure.
- Worker/scheduler startup issues -> visible through run gaps + system health signals.
