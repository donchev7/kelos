# Moontide AI - Thread-Scoped Agent Runtime Spec

**Date:** 2026-02-20  
**Status:** Draft for planning

---

## 1. Concise Business Objective

Ship a Slack-first, real-integration E2E experience where each new root-level app mention creates one shared AI teammate for that thread:

1. One Slack thread maps to one persistent OpenCode session in one E2B sandbox.
2. All human participants in that thread collaborate with the same agent context.
3. Agent outputs stream back into that same Slack thread in near real time.
4. No backend-templated repo-choice prompts or string-matching reply logic.
5. Users can explicitly end the session with an in-thread command.

Success means teams can treat a Slack thread like a multiplayer agent workspace with deterministic repo access boundaries.

---

## 2. Scope

### In scope

1. Thread-scoped session lifecycle (`created -> active -> idle -> closed`).
2. Root mention starts session; all thread replies are routed to same session.
3. Streaming agent output into Slack thread.
4. Deterministic, allowlisted multi-repo access per agent.
5. Explicit terminate command to end sandbox/session.
6. Idle timeout that only applies when agent is not actively running.
7. Real integrations only (Slack, GitHub, E2B, OpenAI/OpenCode).

### Out of scope

1. Vector indexing or long-term semantic memory.
2. Background autonomous jobs outside thread-driven interaction.
3. Multi-agent collaboration in a single thread.
4. Fine-grained per-user ACL inside one Slack thread.

---

## 3. Product Behavior Contract

### 3.1 Session creation

1. A new session is created only from a root-level channel mention or DM root message that mentions the app.
2. Session key is deterministic:
   - Channel root mention: `workspace_id + channel_id + root_ts`
   - DM root mention: `workspace_id + channel_id + root_ts`
3. First agent response in-thread is generated by OpenCode, not templated by backend.

### 3.2 Session participation

1. Any member who can post in that thread can interact with the agent.
2. Subsequent messages in the same thread do not require re-mention.
3. Every human message in-thread is forwarded to the same session queue.
4. Messages from bot users (including app self messages) are ignored for input.

### 3.3 Repo handling

1. Agent configuration owns the allowlisted repos.
2. On session start, backend materializes allowed repos in sandbox workspace.
3. Sandbox only receives credentials and access necessary for those repos.
4. Backend does not emit repo-selection template text.
5. OpenCode can ask natural clarifying questions when ambiguity exists.

### 3.4 Agent output and streaming

1. Agent output is streamed to Slack thread as it is produced.
2. Partial output is posted via throttled message updates.
3. Final output is posted as completed assistant turn in the same thread.
4. First cut streams all OpenCode events into Slack thread as compact text updates (including tool activity and permission prompts).
5. Permission prompts must be visible in-thread and tied to a specific pending permission request ID.

### 3.5 Session termination

1. In-thread command `@App terminate` closes session.
2. Close operation cancels active run, tears down sandbox, marks session closed.
3. Closed sessions reject further thread messages with a short instruction to start a new root mention.

## 4. Functional Requirements

### FR-01 Slack event routing

1. Ack Slack events within 3 seconds.
2. Deduplicate by `event_id + workspace_id`.
3. Route to thread session by root timestamp.
4. Maintain FIFO queue per session to serialize concurrent human turns.
5. Ignore unsupported subtypes, retries already processed, and bot-originated messages.

### FR-02 Session manager

1. Persist `thread_sessions` record keyed by workspace/channel/root_ts.
2. Persist `sandbox_id`, `agent_runtime_session_id`, `status`, timestamps.
3. Enforce single active run per thread session.
4. Expose operations:
   - `createOrGetSession`
   - `enqueueTurn`
   - `startRun`
   - `appendStreamChunk`
   - `finishRun`
   - `cancelAndClose`

### FR-03 Sandbox runtime

1. Start one E2B sandbox per thread session.
2. Prepare workspace with allowlisted repos only.
3. Start one OpenCode conversation context bound to the thread session.
4. Reuse same sandbox and conversation across turns until closed.
5. Support debug mode TTL override (for inspectability) without changing production defaults.
6. Runtime must use OpenCode session APIs (`session.create`, `session.prompt`) plus event subscription, not one-shot CLI mode, so turn streaming and permission handling work.

### FR-04 Streaming bridge to Slack

1. Create one placeholder reply message for each run.
2. Subscribe to OpenCode event stream for the active session/run and mirror events into Slack.
3. Stream content via periodic `chat.update` (rate-limited).
4. Permission request events are posted immediately with the permission request ID and expected reply commands.
5. Tool lifecycle events are also streamed in first cut as compact lines.
6. On completion, finalize message and persist transcript metadata.
7. On failure, post concise error in-thread and keep session usable unless fatal.

### FR-05 Command handling

1. Recognize explicit control commands in-thread:
   - `@App terminate`: terminate thread session and sandbox
   - `@App approve`: approve most recent pending permission request for the thread
   - `@App deny`: deny most recent pending permission request for the thread
2. Commands are parsed before forwarding message to model.
3. Non-command text is always treated as model input.

### FR-06 Idle timeout policy

1. Default idle timeout is 30 minutes.
2. Idle means all of the following are true:
   - no queued human turns for the thread
   - no active model generation
   - no running tool execution
3. A pending permission request is treated as idle wait state (the agent is blocked, not running).
4. Any inbound human message or outbound agent event resets idle clock.
5. Session closes when idle threshold is exceeded.
6. Timeout remains environment-configurable, with 30 minutes as default.

### FR-07 Observability

1. Correlate Slack event, session, sandbox, and model run IDs in logs.
2. Emit structured lifecycle logs for create, run start, stream, complete, timeout, terminate, failure.
3. Store minimal run metadata for debugging and replay safety.

## 6. State Machine

1. `active`: session exists and can accept turns.
2. `running`: a turn is currently executing (can be represented as `active` + run lock).
3. `idle`: no active run, waiting for input.
4. `closing`: teardown in progress.
5. `closed`: final state; no additional turns accepted.
6. `error`: terminal or recoverable error state depending on policy.

Allowed transitions:
1. `active -> idle` after run completion.
2. `idle -> active` on next queued turn.
3. `active|idle -> closing` on terminate/timeout/fatal error.
4. `closing -> closed` on successful teardown.

---

## 7. Security and Boundary Rules

1. Repo access must stay limited to agent allowlist.
2. No wildcard repo escalation at runtime.
3. Secrets remain server-side where possible; sandbox gets only scoped, short-lived credentials.
4. Slack responses are confined to originating thread.
5. All command actions (`terminate`, `approve`, `deny`, and future admin commands) require thread context checks.

---

## 8. Failure Handling

1. Slack delivery failure retries with bounded backoff.
2. Sandbox startup failure posts explicit thread error and marks session `error` or `closed`.
3. Model/tool failure posts concise failure and keeps session open unless repeated fatal threshold is exceeded.
4. Deduplication prevents double-processing from Slack retries.

---

## 9. Acceptance Criteria (E2E)

1. Root mention in channel creates new thread-scoped session and returns first non-templated agent response.
2. Two or more humans can converse with the same agent in same thread without re-mention.
3. Agent responses stream into thread during generation.
4. Agent answers can use any allowlisted repo without backend repo-template prompts.
5. `@App terminate` in-thread terminates session and sandbox.
6. Post-terminate thread messages are rejected with deterministic guidance to start a new root mention.
7. Idle timeout closes inactive session while never interrupting an actively running turn.
8. Permission requests from OpenCode are streamed into the thread and can be approved/denied in-thread.

---

## 10. Decisions Locked For This Cut

1. Termination command is `@App terminate`.
2. Idle timeout default is 30 minutes.
3. `status` command is out of scope.
4. Stream all OpenCode events in first cut, including permission prompts.
